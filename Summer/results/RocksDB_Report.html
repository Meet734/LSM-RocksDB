Compression Codec Winner: lz4

Read Throughput vs DB Size lz4 base default

⚡ Key Notes & Behavior

  • lz4 & snappy dominate: Shrink DB by ~40% (1,419MB → 850MB) AND increase read throughput over uncompressed.
  • Why smaller is faster: I/O-bound workload; smaller SST files keep more of the active working set resident in OS page cache.
  • zstd trap at 1KB: Heavy CPU decompression overhead is not repaid by extra space savings at 1KB value sizes.

Memtable Size Write-Amp vs Space-Amp

Flush Count (Bars) vs Read Ops/s (Line) write_buffer_size

⚡ Key Notes & Behavior

  • Flush reduction: Doubling memtable size halves flush count (128 → 64 → 31 → 16), sharply reducing write amplification.
  • Read ops climb: Throughput rises (+12% at 512MB) because fewer flushes leave fewer overlapping SST files to merge.
  • The 512MB Trap: Resulting DB size doubles to 1,786 MB (~2× space amp)! Huge buffers defer compaction, leaving unmerged obsolete keys on disk.

Parameter Sweep 4 · 8M Records

Block Cache Size Saturates Early

Read Throughput vs Block Cache Size working set ≈ 850 MB

⚡ Key Notes & Behavior

  • Flat curve above 1GB: 512MB is slightly slow due to cache misses. Once cache reaches ~1GB, the entire ~850MB working set fits in memory.
  • Zero gain at 2GB / 4GB: Allocating more RAM buys nothing; minor fluctuations are run-to-run noise and OS page cache competition.
  • Rule of thumb: Size block cache to match working set size. Don't over-allocate RAM here when bgjobs or memtable tuning yield bigger returns.

Parameter Sweep 5 · 8M Records

Direct I/O Toggle 4.8× Read Penalty

Buffered vs Direct — Read Ops/s (Bars) & Read p99 µs (Line) use_direct_reads

⚡ Key Notes & Behavior

  • Decisive failure on drvfs: Enabling Direct I/O collapses random read throughput from 448k down to 93k ops/s (-79%)!
  • Latency explosion: Read p99 tail latency skyrockets from 9 µs to 215 µs (24× worse).
  • Why it happens: Direct I/O bypasses the OS page cache. On WSL2 `/mnt/d`, that warm OS page cache WAS the primary read performance engine. Forcing physical I/O on every read is punishing.

Parameter Sweep 6 · 8M Records

Compaction Style Level vs Universal

Read Throughput (ops/s) & DB Size (MB) FIFO excluded (mirage)

⚡ Key Notes & Behavior

  • Level Compaction: Balanced default. Fully compacted DB (860MB) and fastest legitimate reads (443k ops/s).
  • Universal Compaction: Reduces write amplification, but incurs ~2× space amplification (1,683MB) and 12% slower reads.
  • FIFO Mirage (4.6M ops/s): NOT real performance! FIFO drops 96% of keys by age (only 37,220 / 1,000,000 keys survived). Reads hit bloom filters over a 38MB stub DB.

Scaling Wall · Tail Latency

Tail Latency Climbs with Dataset Size p99 Triples

Read p99 Latency (µs) vs Record Count Lower is better

⚡ Key Notes & Behavior

  • p99 roughly triples: For lz4, tail latency rises from 10 µs @ 8M to 35 µs @ 100M records.
  • Cache miss penalty: Each miss in page cache forces a physical disk seek. As misses become frequent past the RAM boundary, tail latency degrades first.
  • Early warning indicator: Tail latency moves inversely with throughput; monitoring p99 provides an early warning before throughput collapses.

YCSB Workload A · Thread Scaling

Thread Scaling Inverts Under Disk Pressure Non-Obvious

Workload A Throughput by Thread Count (4 vs 8 vs 16) 1M → 10M → 50M

⚡ Key Notes & Behavior

  • 1M (RAM Cached): More threads help! 4→8→16 threads scale from 5.8k to 7.8k ops/s (CPU bound).
  • 10M (Inversion Point!): 4 threads (3,506 ops/s) BEATS 8 threads (2,584 ops/s)! Why? High concurrent random I/O and compaction contention on a single WSL2 disk causes thrashing.
  • 50M (Heavy Disk): 16 threads recover slightly (1,111 vs 940 ops/s) by hiding per-request disk latency.
  • Core Takeaway: Optimal thread count is dynamic and scale-dependent! A fixed thread pool is sub-optimal across scales.

Write-Path & Stalls · 8M → 100M Records

Write-Stalls Explode 44× at Scale Super-Linear Debt

Compaction Time (ms) & Write-Stall Time (µs) vs Scale lz4 base config

⚡ Key Notes & Behavior

  • 12.5× Data vs 44× Stalls: From 8M to 100M records, data grows 12.5×, compaction time tracks it (12.5×), but write stalls explode ~44× (54ms → 2.4s)!
  • Why stalls compound: Flushes queue up faster than background threads can compact deeper LSM levels, forcing RocksDB to stall incoming writes to prevent space exhaustion.
  • Mitigation: Bigger memtable (256M/512M) or more background jobs (8–16) are critical to absorb write bursts without user-visible stalls.

Tuning Guidance · Synthesis

Optimal Starting Config & Knob ROI Where to Spend Effort

Read Throughput Swing (%) Produced by Each Knob (Best vs Worst in 8M Sweep) ROI Ranking

⚡ Priority ROI Ranking

  • 1. Direct I/O OFF (382% swing): Single biggest lever. Never bypass OS page cache on cache-backed storage.
  • 2. Background Jobs 8–16 (39% swing): Free performance boost; match to available CPU cores.
  • 3. lz4 / snappy Compression (20% swing): Shrinks DB and speeds up reads simultaneously.
  • 4. Memtable & Cache (13–14% swing): Second-order. Set memtable to 256MB and cache to fit working set.
